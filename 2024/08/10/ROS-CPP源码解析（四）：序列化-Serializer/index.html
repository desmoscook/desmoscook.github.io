ROS

上节我们提到，使用 Stream 进行传输数据的时候，实际上会调用 Serializer 类的成员进行序列化，这也是这一章的重点。

但是在介绍 Serializer 类之前，我们先来介绍一下序列化的消息类 SerializedMessage ，序列化和反序列化的结果都会存储在这个类中。

SerializedMessage 类

SerializedMessage 是 ROS 种用于消息序列化和反序列化的类，其中同时包含 序列化 所需要的数据成员和 反序列化 所需要的数据成员

如果消息被序列化，那么 SerializedMessage 的 buf 成员应该不为 null
如果消息被反序列化，那么 SerializedMessage 的 message 成员应该不为 null

在 pub流程 章节中，Publication 就是根据 SerializedMessage 的成员来判断当前的数据是否序列化，从而判断是进程内通信还是进程间通信的。

这里需要理解 message_start 和 buf.get() 两个指针的区别，使用图示表示 序列化后的message 如下。

由于读取是直接读取字节流，因此我们不能分辨一个 message 的消息到底从哪里结束，所以对于每一个被序列化的消息，都会在前4个字节存储整个消息的长度。

buf.get() 指向的是整个消息的起始，而 message_start 指向的是 正真被存储的数据 的起始。

Serializer 类

由于 Stream 的操作，实际上都是 Serializer 这个幕后老大在实现。我们具体看一下 Serializer 类是如何实现的。

这里可以看到 Serializer 类调用的实际上是 传入的 Message 的成员函数作为序列化方法。那内置类型应该如何序列化？内置类型（如int）可不会提供这些成员共 Seriaizer 使用。

实际上，ROS 通过模板的全特化实现了基本类型的序列化。

基本类型序列化

对于这些常见的基础类型 int ，float， double 等，ROS 实际上并没有进行序列化，而是直接传输。

std::string

对于 string 的序列化，由于在流中无法确定边界，所以使用固定的 4 个字节来存储 string 的长度。

在 string 的反序列中，首先会取出前 4 个字节，然后根据其中存储的 size 来确定后续多少字节属于当前的 string

ROS 实际上对于大量的类型进行了全特化来支持，如 ros::Time，ros::Duration 等等，这里就不再赘述

serializeMessage 和 deserializeMessage 函数

回顾一下，roscpp_serialization 主要包含的 4 个类我们已经讲解完毕，分别是

  • SerializedMessage
  • Serializer
  • Stream
  • IStream和OStream

这 4 个类实际上都是内部类，我们不会直接操作这些类进行序列化，ROS 提供了统一的接口用来实现序列化，即 serializeMessage() 和 deserializeMessage() 函数

在 pub 和 sub 的流程中，实际上得到 message 之后，只需要调用这个函数就会进行消息的序列化和反序列化。

这两个函数的源码如下，相信如果了解了这些内部类的成员之后，理解这个外部接口应该不是很困难。

对于 serializeMessage() 函数

  1. 首先计算出了 message 序列化后的长度 len，并将 buf 设置为 len + 4。这里多出来的4个字节用于存储该 message 在传输中的总长度，用于流的解包
  2. 然后使用 OStream 对 buf 进行操作，先将 num_bytes 序列化并输入，共占用 4 个字节
  3. 最后序列化 message 本身

对于 deserializeMessage() 函数

  1. IStream s(m.message_start, m.num_bytes - (m.message_start - m.buf.get())) 中
  2. 如果消息包含长度，就会根据 m.message_start - m.buf.get() 计算出长度的大小并跳过

通过这两个函数，我们就可以轻松的序列化数据和反序列化数据了。
